# Bug List - 2026-05-15

## Bug 1: web_fetch 工具在异步上下文中触发 tokio runtime panic

- **解决时间**: 2026-05-15 09:30
- **模块**: `crates/core/src/tools/basic_tools.rs`
- **现象**: 当 Agent 在流式响应中调用 `web_fetch` 工具时，程序 panic：
  ```
  thread 'tokio-rt-worker' panicked at .../tokio-1.51.1/src/runtime/blocking/shutdown.rs:51:21:
  Cannot drop a runtime in a context where blocking is not allowed.
  This happens when a runtime is dropped from within an asynchronous context.
  ```
- **根因**: `BasicTool::run_web_fetch()` 直接调用了 `reqwest::blocking::get()`。`reqwest::blocking` 模块内部会隐式创建一个 tokio runtime 来驱动异步 HTTP 请求。当 `run_web_fetch` 在 Agent 的异步循环（tokio runtime）中被调用时，`reqwest::blocking` 创建的嵌套 runtime 在函数返回后被 drop，而 drop 操作发生在异步上下文中，触发 tokio 的 panic 保护机制。
- **修复方案**: 将 `reqwest::blocking::get()` 的调用包裹在 `std::thread::spawn()` 中，在新线程中执行 blocking HTTP 请求，避免在 tokio worker 线程上创建/销毁嵌套 runtime。

## Bug 2: TUI 消息区域内容紧贴左边框，无 padding 导致阅读体验差

- **解决时间**: 2026-05-15 23:32
- **模块**: `crates/tui/src/components/chat.rs`
- **现象**: AI 消息区域（包括 Thinking、Text、Tool 结果等）的文字内容直接贴在左侧竖线边框上，没有留白，视觉上拥挤且影响阅读。用户消息区域也存在同样的问题（被 "● You" 前缀部分遮挡，不明显）。
- **根因**: `chat.rs` 在实现消息区域背景色和左边框时，内容区域仅向右偏移了 1 列（留给左边框本身），`total_height()` 和 `draw()` 中的宽度计算均使用 `width.saturating_sub(1)`。这导致内容从边框右侧第 1 列就开始渲染，中间没有任何空白 padding。
- **修复方案**:
  1. AI 内容区域 `clip_rect_ai` 的 x 偏移从 `inner.x + 1` 改为 `+ 2`，宽度从 `inner.width - 1` 改为 `- 2`
  2. 用户消息区域 `content_rect` 同步改为 `x + 2`、`width - 2`
  3. `total_height()` 中所有 AI part 和用户消息的 `line_count` 宽度统一从 `saturating_sub(1)` 改为 `saturating_sub(2)`，确保滚动高度和实际渲染区域严格一致
