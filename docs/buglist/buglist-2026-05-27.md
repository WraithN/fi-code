# Bug List 2026-05-27

## 解决时间：13:38

### 模块：agent / tools / frontend

### 现象：
1. Web 模式下 `ask_for_question` 工具被调用时，其原始参数 JSON（包含 `question`、`options`、`recommended` 等字段）被直接展示在聊天记录中。
2. 问题弹窗（`PermissionDialog`）虽然触发，但可能因为原始 JSON 的干扰导致用户体验差。用户未在 60 秒内回答，触发超时错误。
3. `PermissionDialog` 使用 `absolute bottom-full` 定位，内容较多时向上延伸，顶部可能被浏览器视口截断。

### 根因：
1. **后端 `agent.rs` 和 `runner.rs`**：在 LLM 流式输出和工具执行前，会批量/实时发送 `SseEvent::Part { part: ToolUse { ... } }` 给前端。对于 `ask_for_question` 这种交互式工具，其参数 JSON 本就不应直接展示给用户，但后端没有过滤。
2. **前端 `ToolUsePart`**：直接渲染所有 `ToolUse` 的 `arguments` JSON，没有对 `ask_for_question` 做特殊处理。
3. **前端 `PermissionDialog` 定位**：`absolute bottom-full` 让弹窗底部对齐父容器顶部，内容多时向上生长，容易超出视口顶部被截断。

### 修复方案：
1. **后端跳过 `ask_for_question` 的 ToolUse SSE 发送**：
   - `agent.rs` line 571-588：在批量发送 `ToolUse` 事件时，如果 `name == "ask_for_question"`，直接 `continue` 跳过。
   - `runner.rs` line 193-199：在流式输出实时发送 `ToolUse` 时，用 `matches!` 过滤掉 `ask_for_question`。

2. **前端 `ToolUsePart` 兜底不渲染**：
   - `ToolUsePart.tsx`：如果 `part.name === 'ask_for_question'`，返回 `null`，避免任何情况下展示原始参数 JSON。

3. **前端 `PermissionDialog` 定位优化**：
   - 将 `absolute bottom-full` 改为 `fixed bottom-20`，确保弹窗始终停留在视口底部上方，不受容器滚动或截断影响。
   - 添加 `max-h-[60vh] overflow-y-auto`，防止内容过多时向上延伸超出视口。

### 相关文件：
- `crates/core/src/agent/agent.rs`
- `crates/core/src/agent/runner.rs`
- `frontend/src/components/part-renderers/ToolUsePart.tsx`
- `frontend/src/components/PermissionDialog.tsx`

- fix: PermissionDialog 改为 fixed 定位避免被截断

## 解决时间：15:49

### 模块：server/api/permission_api.rs, frontend/services/apiClient.ts

### 现象：
用户点击 `ask_for_question` 弹窗中的选项后，前端 `PermissionDialog` 抛出错误：
```
[PermissionDialog] Failed to respond question: Error: API returned no data
    at ApiClient.post (apiClient.ts:58:52)
```
导致用户回答无法送达后端，`wait_question_response` 最终因 60 秒超时而失败。

### 根因：
**前后端 API 响应语义不一致。**

1. **后端** `handle_question_respond` 和 `handle_permission_respond` 在操作成功后，返回 `ApiResponse::success(Value::Null)`，生成的 JSON 为：
   ```json
   { "success": true, "data": null, "error": null, "error_code": null }
   ```
   这在 Rust 语义中表示"操作成功，无返回数据"，是合理的 REST 设计。

2. **前端** `ApiClient.post<T>` 的校验逻辑过于严格（`apiClient.ts:58`）：
   ```typescript
   if (!data.success || data.data === null) throw new Error(data.error || 'API returned no data');
   ```
   该判断将 `data: null` 统一视为错误，即使 `success: true` 也无法通过。`serde_json::Value::Null` 序列化后即为 JSON `null`，因此 `data.data === null` 为 `true`，触发异常抛出。

### 修复方案：
**方案 A：后端返回非空标记对象（已采纳）。**

将 `ApiResponse::success(Value::Null)` 改为返回一个非空的 JSON 对象，确保 `data` 字段不为 `null`：

```rust
// permission_api.rs:102
Json(ApiResponse::success(serde_json::json!({ "received": true })))

// permission_api.rs:120
Json(ApiResponse::success(serde_json::json!({ "received": true })))
```

这样前端 `data.data === null` 为 `false`，校验通过，不再抛出错误。

**未采纳的方案 B**：前端放宽校验（`if (!data.success)` 不再检查 `data.data === null`），考虑到 `data: null` 在错误场景下也可能出现，统一放宽可能掩盖其他后端错误，因此选择由后端保证 `data` 非空。

### 相关文件：
- `crates/core/src/server/api/permission_api.rs`

### 相关 Commit：
- fix: 修复 PermissionDialog API error 因后端返回 data: null 导致前端误判为错误

## 解决时间：17:01

### 模块：frontend/tailwind.config.js, crates/core/src/tools/mod.rs

### 现象：
1. 内联权限确认组件（`InteractivePermissionPart`）的"确认执行"按钮鼠标悬停时没有高亮效果。
2. `write` / `edit` 工具的权限确认只显示通用提示 "tool write may modify files, requires confirmation"，未展示具体要操作哪个文件。

### 根因：
1. **Tailwind 未定义 `primary` 和 `primary-hover` 颜色**：`tailwind.config.js` 的 `colors` 扩展中没有 `primary` 和 `primary-hover` 键，导致 `bg-primary` 和 `hover:bg-primary-hover` 类名无法生成对应的 CSS，按钮 hover 状态无视觉变化。
2. **后端未传递文件路径信息**：`tools/mod.rs` 发送 `PermissionAsk` SSE 时，仅使用 `permission.rs` 返回的 `reason` 字符串，未从工具参数（`arguments`）中提取 `path` 等关键信息附加到提示中。

### 修复方案：
1. **前端**：在 `tailwind.config.js` 的 `theme.extend.colors` 中添加：
   ```js
   primary: '#24C8DB',
   'primary-hover': '#3DD4E6',
   ```
   使 `bg-primary` / `hover:bg-primary-hover` 生效，按钮悬停时变为更亮的青色。

2. **后端**：在 `tools/mod.rs` 发送 `PermissionAsk` SSE 之前，从 `arguments` 中提取 `path` 字段，附加到 `reason`：
   ```rust
   let path_info = arguments.get("path")
       .and_then(|v| v.as_str())
       .map(|p| format!(" (目标文件: {})", p))
       .unwrap_or_default();
   let enriched_reason = format!("{}{}", reason, path_info);
   ```
   这样前端显示的提示变为 "tool write may modify files, requires confirmation (目标文件: src/main.rs)"。

- `frontend/tailwind.config.js`
- `crates/core/src/tools/mod.rs`

## 解决时间：22:20

### 模块：crates/core/src/agent/agent.rs

### 现象：
Agent 在任务完成后陷入死循环，不断重复调用 LLM，input_tokens 持续增长。从 spans.jsonl 可见 `fi_code.turn.index` 始终停留在 3，且每轮请求的 `gen_ai.usage.input_tokens` 递增（13892 → 14010 → 14125 → ...），说明历史消息不断累积但轮次不前进。

### 根因：
**任务锚点校验标志与 `run_one_turn` 内部的 `transition_reason` 发生覆盖冲突。**

1. `agent_loop` 在 `run_one_turn` 返回 `false`（任务结束）后，会检查 `state.transition_reason != Some("intent_verified")`，若未校验则插入校验消息并 `continue`。
2. `run_one_turn` 的 **direct_output 优化分支**（line 672）在返回前设置了 `state.transition_reason = Some("direct_output".to_string())`。
3. 当一轮对话触发 direct_output（所有工具成功且 Turn 1 有前置文本）时：
   - 第一次回到 `agent_loop`：`transition_reason` 是 `"direct_output"`，不等于 `"intent_verified"`，于是插入校验消息，`continue`。
   - 第二次调用 `run_one_turn`：再次触发 direct_output，`transition_reason` 被覆盖为 `"direct_output"`。
   - 第二次回到 `agent_loop`：`transition_reason` 仍然是 `"direct_output"`，再次不等于 `"intent_verified"`，再次插入校验消息，`continue`。
   - 由此进入 **无限循环**。

### 修复方案：
将"任务锚点是否已校验"从 `transition_reason` 中分离，使用独立的布尔标志 `intent_verified`：

1. **在 `LoopState` 新增字段**：
   ```rust
   pub intent_verified: bool,
   ```
   并在 `LoopState::new()` 中初始化为 `false`。

2. **修改 `agent_loop` 校验逻辑**：
   ```rust
   // 修复前：依赖 transition_reason（会被 run_one_turn 覆盖）
   if state.transition_reason != Some("intent_verified".to_string()) { ... }

   // 修复后：使用独立标志
   if !state.intent_verified {
       ...
       state.intent_verified = true;
       continue;
   }
   ```

3. **同步更新 `server/session.rs`**：
   在 `HttpSessionManager::get()` 的手动 `LoopState` clone 逻辑中同步复制 `intent_verified` 字段。

### 相关文件：
- `crates/core/src/agent/agent.rs`
- `crates/core/src/server/session.rs`

### 相关 Commit：
- fix: 修复任务锚点死循环——intent_verified 与 transition_reason 分离

## 记录 3：重构 read / glob / grep 工具——以 .gitignore 替代硬编码黑名单

**解决时间：** 2026-05-27 22:35

**模块：** `crates/core/src/tools/basic_tools.rs`、`crates/core/src/tools/mod.rs`

**现象：**
- `BLOCKED_DIRS` 硬编码了 30 个目录名（如 `target`、`node_modules`、`dist`、`build` 等），存在两个问题：
  1. **误伤**：`out`、`dist`、`build` 等可能是合法源码目录，却被无条件排除；
  2. **僵化**：不同项目的排除规则不同，硬编码列表无法自适应。

**根因：**
早期的文件搜索工具（grep、glob）使用静态黑名单过滤，没有利用项目已有的 `.gitignore` 文件来动态决定排除范围。

**修复方案：**

1. **引入 `ignore` crate**：在 `crates/core/Cargo.toml` 中添加 `ignore = "0.4"`，用于解析 `.gitignore` 规则。

2. **read 工具增强**：
   - 在 `limit` 基础上新增 `offset` 参数（1-based），支持从指定行号开始读取；
   - 当 `offset > 1` 时，输出头部添加 `... (N lines skipped)` 提示；
   - 当内容被截断时，尾部添加 `... (N more lines)` 提示。

3. **grep 工具重构**：
   - **移除 `BLOCKED_DIRS` 常量** 和 `is_blocked_path` 辅助函数；
   - **rg 路径**：移除 `--glob` 排除规则和 `--no-hidden` 参数（rg 默认会读取 `.gitignore` 并跳过隐藏文件）；
   - **fallback 路径**：将 `walkdir::WalkDir` 替换为 `ignore::Walk`，自动尊重 `.gitignore`；保留隐藏文件跳过逻辑（与 rg 默认行为一致）。

4. **glob 工具重构**：
   - **优先使用 `rg --files`**（如果 ripgrep 可用）：自动尊重 `.gitignore`，性能更好；
   - **fallback**：使用 `ignore::Walk` 遍历目录树，再用 `glob::Pattern` 过滤匹配文件；
   - 统一保留隐藏文件跳过逻辑。

5. **Schema 更新**：更新 `read` 工具的 JSON Schema，加入 `offset` 参数描述。

6. **测试更新**：
   - `test_grep_skips_blocked_dirs` → `test_grep_respects_gitignore`：创建临时 `.gitignore` 文件验证排除行为；
   - 新增 `test_run_read_with_offset`：验证 offset + limit 的组合行为。

**相关文件：**
- `crates/core/Cargo.toml`
- `crates/core/src/tools/basic_tools.rs`
- `crates/core/src/tools/mod.rs`

**相关 Commit：**
- refactor: 以 .gitignore 替代 BLOCKED_DIRS，增强 read/glob/grep 工具
